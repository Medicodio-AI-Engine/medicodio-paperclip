# SEO Content Analysis Skill

**Purpose:** Analyse a raw markdown blog draft for content quality, GEO/AI citation readiness, and schema generation. Operates on provided text — no URL fetching, no external tools. Works identically on Paperclip hosted agents and local Claude Code.

**Paperclip skill keys (GitHub managed):** This skill adapts the following Paperclip-managed skills for raw markdown text input:
- `seo-content` — Content Quality & E-E-A-T
- `seo-geo` — GEO / AI Citation Readiness
- `seo-schema` — Schema Markup Generation

When these skills are attached to the agent in Paperclip, the agent may use their full logic. The instructions below are the text-adapted version that works without a live URL.

---

## Inputs

The calling phase must supply these values before invoking this skill:

| Input | Source |
|-------|--------|
| `draft_content` | Draft markdown text in memory (post any auto-fixes, excluding frontmatter `---` block) |
| `topic` | From run-state.json `topic` field |
| `primaryKeyword` | From run-state.json `primaryKeyword` field |
| `seoTitle` | From draft frontmatter `seoTitle` field |
| `seoDescription` | From draft frontmatter `seoDescription` field |
| `publishedAt` | From draft frontmatter `publishedAt` field |

---

## Outputs

Capture all values below into memory before returning to the calling phase:

| Output | Type | Description |
|--------|------|-------------|
| `content_quality_score` | integer 0–100 | E-E-A-T total score |
| `eeat_breakdown` | object | `{ experience: N, expertise: N, authoritativeness: N, trustworthiness: N }` |
| `content_quality_flags` | list | AI content marker issues found (empty = clean) |
| `ai_tone_score` | float 0–10 | LLM-judged AI tone average across H2 sections (lower is better; >7 = flagged) |
| `ai_tone_worst_sections` | list | Top 3 sections by AI tone score desc, each `{section_title, section_index, current_snippet, score, justification}` — consumed by seo-improve.md for targeted rewrites |
| `internal_link_opportunities` | list | `[{ anchor_text, target_topic }]` |
| `geo_score` | integer 0–100 | GEO / AI citation readiness total |
| `geo_top5` | list | Top 5 GEO improvements ordered by impact |
| `geo_passages_to_restructure` | list | `[{ current_snippet, suggested_version }]` |
| `schema_json` | string | BlogPosting JSON-LD ready for publish payload |

## Part 1 — Content Quality & E-E-A-T Analysis

*Implements: `seo-content` skill adapted for raw markdown text*

### 1a. Word Count
Count words in `draft_content`. Blog floor: 1,500 words. Flag as `BELOW FLOOR` if under.

### 1b. Readability
Scan `draft_content` for:
- **Sentence length:** Average target 15–20 words. Flag if consistently above 25.
- **Paragraph length:** Target 2–4 sentences. Flag paragraphs with > 6 sentences.
- **Heading hierarchy:** Verify H1 → H2 → H3 with no skipped levels.
- **Table of contents:** Required for drafts > 1,500 words. Flag if absent.
- **Lists:** Multi-item content (3+ items) should use bullets or numbered lists, not run-on prose.

### 1c. E-E-A-T Scoring (Sept 2025 QRG + Dec 2025 core update)

Score each dimension 0–25. Deductions floor at 0 per dimension.

**Experience (0–25)** — first-hand knowledge:
- Original case studies or real-world examples with specific details: +8
- Original statistics, data, or proprietary insights cited in draft: +8
- Step-by-step process documentation showing real methodology: +5
- Personal anecdotes that could not be generated by a generic AI: +4

**Expertise (0–25)** — depth and credentials:
- Author byline with relevant credentials visible in draft body: +8
- Technical accuracy and depth (assess: is this surface-level or substantive?): +8
- Claims supported by cited sources (inline references, footnotes, or linked studies): +5
- Correct use of specialized vocabulary throughout: +4

**Authoritativeness (0–25)** — external recognition embedded in content:
- Links or citations to authoritative sources (official bodies, academic studies, industry data): +10
- Industry recognition, certifications, or affiliations mentioned: +8
- Consistent topical focus — no drift to unrelated subjects: +7

**Trustworthiness (0–25)** — transparency:
- `publishedAt` date populated in frontmatter: +10
- Author or organization contact information referenced in draft: +8
- No clickbait, false urgency, or misleading claims detected: +7

Set `content_quality_score = experience + expertise + authoritativeness + trustworthiness`.
Set `eeat_breakdown = { experience: N, expertise: N, authoritativeness: N, trustworthiness: N }`.

### 1d. AI Content Quality Markers (rule-based + LLM-judged)

**Pass A — pattern matching (deterministic):**
Scan `draft_content` for these low-quality signals. Each instance found goes into `content_quality_flags`.

Flag:
- Filler openers: "In today's world", "It is important to note", "It goes without saying", "Needless to say"
- Vague citations without attribution: "many studies show", "experts agree", "research indicates" (no source given)
- Identical sentence-opening patterns across 3+ consecutive paragraphs
- Overused transitions: "Additionally", "Furthermore", "Moreover" appearing more than 3× per 500 words
- No first-hand signals anywhere in the draft (entire content reads as assembled from public sources)
- No author attribution in the draft body

**Pass B — LLM-judged AI tone (subjective):**
For each H2 section in `draft_content` (split on `## ` heading lines), prompt the model:

> "Read this paragraph from a B2B blog post. Score 0–10 how AI-generated it reads. 0 = clearly written by a human practitioner with first-hand voice, specific anecdotes, opinionated claims. 10 = obvious AI tone — generic structure, balanced both-sides framing, listicle format, no personal voice, vague universal claims. Reply with a single integer 0–10 and a one-sentence justification.
>
> Paragraph:
> {section_text}"

Capture each section score. Compute:
```
ai_tone_score = mean(section_scores)        # 0–10, lower is better

ai_tone_worst_sections = top 3 sections by score desc, each as:
{
  section_title:    H2 heading text (without the "## " prefix),
  section_index:    0-based index of the section in document order,
  current_snippet:  first 25 words of the section body,
  score:            integer 0–10 (the LLM's score for this section),
  justification:    the LLM's one-sentence rationale
}
```

Aggregate:
- IF `ai_tone_score > 7` → append `"AI tone: heavy ({ai_tone_score:.1f}/10) — sections [{section_titles}] flagged for rewrite"` to `content_quality_flags`.
- IF `ai_tone_score > 8` → seo-improve.md Step 3b.B will rewrite the top sections in `ai_tone_worst_sections` with first-hand voice (specific numbers, named tools, practitioner perspective). Do NOT mutate `geo_passages_to_restructure` here — keep these signals separate so seo-improve can apply different rewrite strategies.

`content_quality_flags = []` means no issues found — this is good. Always set `ai_tone_score` (even when no flag) so callers can record it. Always set `ai_tone_worst_sections` to a list (possibly empty if `ai_tone_score <= 7`).

### 1e. Internal Linking Opportunities
Identify 3–5 noun phrases in `draft_content` that would make strong internal link anchors. For each produce `{ anchor_text: "...", target_topic: "what the linked destination page covers" }`. Focus on: product or feature names, technical concepts, topic cluster terms, comparison terms.

Set `internal_link_opportunities` to this list.

## Part 2 — GEO / AI Citation Readiness

*Implements: `seo-geo` skill adapted for raw markdown text (skips live URL checks: robots.txt, llms.txt, SSR detection)*

Score each dimension 0–25 by inspecting `draft_content`.

### 2a. Citability (0–25)
Optimal AI citation passage length: 134–167 words.

- Identify self-contained answer blocks (paragraphs that stand alone without surrounding context). Award +8 if ≥ 3 such blocks exist.
- Check: does each major H2 section open with a direct answer in first 40–60 words? Award +7 if ≥ 3 sections pass.
- Check: are specific statistics or data points cited with source attribution? Award +10 if ≥ 2 such citations exist.
- Deduct 3 per vague general statement lacking data or attribution (max deduction −9).

### 2b. Structural Readability (0–25)
- Question-based H2 or H3 headings (e.g., "How Does X Work?", "What Is X?"): +8 if ≥ 3 present
- Paragraphs of 2–4 sentences throughout (spot-check 5 random paragraphs): +7 if ≥ 4/5 pass
- Comparison or data table present in draft: +5
- FAQ section with question headings and direct answers: +5

### 2c. Authority Signals in Draft (0–25)
- Author byline with credentials: +8
- Inline external citations to authority domains (`.gov`, `.edu`, peer-reviewed journals — NEJM, JAMA, AHIMA, AAPC, CMS, HHS, NIH, AMA): +7
  - Detect by scanning markdown links `[text](url)` for matching domain patterns
  - At least 1 such citation = +7
- At least 2 external citations to primary sources (government, academic, industry body): +10
  - Same detection method, threshold of 2 distinct authority-domain links

Note: `publishedAt` populated is a **Trustworthiness** signal scored in Part 1c. It is intentionally not double-counted here. Authority is about external recognition embedded in the draft body.

### 2d. Answer-First Formatting (0–25)
- Primary topic definition ("X is..." or "X refers to...") appears in first 80 words of the draft body: +10
- At least 2 H2 sections open with the answer in the first sentence before elaborating: +8
- Process explanations use numbered steps rather than prose-only: +7

Set `geo_score = citability + structural_readability + authority_signals + answer_first`.

Identify top 5 highest-impact GEO changes and set `geo_top5` (ordered, most impactful first).

Identify up to 3 existing paragraphs that would benefit from restructuring into 134–167 word self-contained answer blocks. For each, produce `{ current_snippet: "first 25 words of the paragraph...", suggested_version: "rewritten 134–167 word block" }`. Set `geo_passages_to_restructure` to this list.

## Part 3 — Schema Markup Generation

*Implements: `seo-schema` skill for BlogPosting type — generates from draft metadata, no live page needed*

Generate a valid `BlogPosting` JSON-LD block using inputs provided. Use only values available — mark missing values with `{placeholder}` so the publish phase can resolve them.

Extract from inputs:
- `headline` ← `seoTitle`
- `description` ← `seoDescription`
- `datePublished` / `dateModified` ← `publishedAt`
- `keywords` ← `primaryKeyword` + any secondary keywords identifiable from `draft_content` topic context (up to 3 additional)

Generate this JSON-LD (fill in known values, leave `{placeholder}` for unknowns):

```json
{
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  "headline": "{seoTitle}",
  "description": "{seoDescription}",
  "keywords": "{primaryKeyword}, {secondary keywords from draft context}",
  "datePublished": "{publishedAt}",
  "dateModified": "{publishedAt}",
  "author": {
    "@type": "Organization",
    "name": "{organization name — from topic context or placeholder}",
    "url": "{site URL — from topic context or placeholder}"
  },
  "publisher": {
    "@type": "Organization",
    "name": "{organization name}",
    "logo": {
      "@type": "ImageObject",
      "url": "{logo URL — placeholder for publish phase}"
    }
  },
  "url": "{canonical URL — placeholder, resolved by publish phase from slug}",
  "mainEntityOfPage": {
    "@type": "WebPage",
    "@id": "{canonical URL — placeholder}"
  },
  "image": {
    "@type": "ImageObject",
    "url": "{featured image URL — placeholder}"
  }
}
```

Set `schema_json` to this JSON-LD as a string.

### Part 3 Scorecard Block (write into log)

```
### Schema Markup — BlogPosting JSON-LD
{schema_json}

Note: Fields marked {placeholder} must be resolved by the publish phase.
```
